# WIL — 프로젝트 전체에서 시도한 접근 방식

> MiniSQL Week 7: B+ Tree 인덱스 구현 스프린트 회고
> 초점: "무엇을 만들었나"가 아니라 **"어떤 문제를 어떤 방식으로 풀려고 했는지, 그 근거가 뭐였는지"**

---

## 0. 출발 문제

기존 MiniSQL 은 `WHERE id = ?` · `WHERE id BETWEEN ...` 같은 질의를 **파일 처음부터 끝까지 읽어서** 처리했다.
1M 건에서는 단건 조회도 수초, 범위 조회도 수 초 단위. 그래서 **B+ Tree 인덱스**로 풀기로 했다.

제약: **1일 스프린트 + 5명 팀 + C 구현 + CSV 저장 기반**.

---

## 1. 아키텍처 — 인터페이스 계약 우선 (Contract-First)

### 시도
- MP1 시점에 `include/bptree.h` 의 6개 함수 시그니처를 먼저 확정하고 **절대 수정 금지**로 고정
- 트리에는 row 데이터 자체를 넣지 않고 **`id → row_index` 정수 매핑만** 저장
- `index_registry.c` 를 도입해 storage 가 등록한 트리를 executor 가 **같은 인스턴스로 조회**하게 중앙 관리

### 근거
- **팀 병렬화:** 계약만 고정되면 bptree(지용) / storage(민철) / executor(정환) 가 서로의 내부를 몰라도 동시에 작업 가능. 1일 스프린트에서 가장 값진 자원은 "대기 시간 0"이었다.
- **저장 형식 독립성:** 트리가 row 를 직접 들고 있으면 CSV → 바이너리 전환 시 트리도 재작성. 매핑만 저장하면 storage 레이어만 바꾸면 된다. (실제로 Round 2에서 바이너리 fast path를 추가할 수 있던 이유)
- **subprocess 모델 대응:** 웹 데모는 각 요청마다 프로세스가 새로 뜬다. `storage_ensure_index` 가 첫 SELECT 진입 시 CSV 스캔으로 트리를 lazy rebuild — "프로세스당 테이블당 1회" 원칙.

### 배운 것
인터페이스 계약을 먼저 고정하는 건 **소규모 팀에서도 유효한 전략**이다. 보통 마이크로서비스 설계에서 나오는 이야기지만, 5명짜리 1일 스프린트에서도 병렬성이 극적으로 올라갔다.

## 2. 성능 — I/O 경계 최적화 패턴의 반복 적용

### 2.1 INSERT 최적화 5단계 (32 ms → 3 µs, 10,000배)

| 단계 | 시도 | 왜 이 방향을 택했나 |
|---|---|---|
| ① 파일 포인터 재사용 | 매 row 마다 open/close → 프로세스 생존 기간 유지 + 64KB setvbuf | **setup 비용을 분산** (TCP keepalive와 같은 원리) |
| ② 스키마 캐싱 | 매 INSERT마다 읽던 스키마 → 첫 1회만 로드 | **안 변하는 데이터는 다시 묻지 않는다** (proxy_cache) |
| ③ 경로 캐싱 | 매번 탐색하던 파일 경로 → 첫 호출에 확정 후 재사용 | **라우팅 결과도 캐시 대상** (resolver cache) |
| ④ meta 캐싱 | 매 INSERT마다 O(N²) 재스캔 → O(1) | **hot path 중복 계산 제거** (microcache) |
| ⑤ BULK_INSERT_MODE | 매번 fflush → 버퍼가 찰 때만 flush | **syscall을 묶어서 지연** (proxy_buffering) |

진짜 중요한 발견: **원인이 알고리즘이 아니라 환경(WSL2 + Docker + 9p 마운트)의 파일 시스템 왕복이었다.** O(log n) 트리를 아무리 잘 짜도 건당 6회의 syscall × 0.3 ms 왕복이 지배하고 있었다.

자세한 분석은 `docs/WIL_insert_reverse_proxy.md` 참고.

### 2.2 바이너리 fast path (CSV 5.77s → 2.22s)
- **시도:** INT/FLOAT/VARCHAR 를 **고정폭 레이아웃**으로 바이너리 저장 → N번째 행을 `fseek` 로 바로 접근
- **근거:** B+ Tree 가 `row_index` 를 반환해도, CSV 는 여전히 N번째 행까지 읽어야 도달 가능 → "트리는 빠른데 storage 가 느린" 병목. 바이너리 고정폭이면 `seek(N × row_size)` 로 O(1) 도달.
- **효과:** 범위 조회 1M 행에서 2.6배 개선. B+ Tree 본래 시간복잡도(O(log n + k))가 비로소 실제 성능으로 드러남.

### 2.3 benchmark.c 로 "자료구조 자체"도 분리 측정
- 전체 SQL 파이프라인 수치 외에, `bptree_search` / `bptree_range` 의 **pure 성능**을 따로 측정
- 근거: 실제 질의 성능에는 파서 + executor + storage 오버헤드가 섞여 있음. 자료구조가 얼마나 기여하는지 분리해서 보여야 "트리 구현 자체가 잘 됐나"를 판단할 수 있음.

### 배운 것
성능 문제는 **"알고리즘 / 시스템 레이어 / 환경 레이어"** 중 어디가 범인인지 측정부터 해야 한다. 그리고 한 번 터득한 최적화 패턴(caching, pooling, batching)은 DB · 웹 서버 · OS · 네트워크에서 **그대로 재사용된다**.

## 3. 협업 — Mix Merge 와 AI Unit Test 위임

### 3.1 Mix Merge
- **상황:** Round 2 에서 정환(executor BETWEEN) 과 민철(storage DELETE/UPDATE 인덱스 동기화) 이 **같은 파일들을 동시에 건드려야** 했다.
- **시도:** 두 PR 각자 PR 본문에 "건드린 함수명"을 명시 → PM 이 충돌 없음 확인 후 **별도 mix 브랜치에서 통합** → 최종 PR 로 dev 에 머지.
- **근거:** 대규모 팀은 rebase-and-merge 로 자동 처리하지만, 5명 스프린트에서는 **사람이 2~3번 들여다보는 게 더 빠르고 안전했다.** 의도를 히스토리에 남길 수 있었고 (`merge: PR #23 ... into mix`, `merge: PR #24 ... into mix`), 충돌 리스크를 PM 한 명이 흡수.

### 3.2 AI Unit Test 위임
- **정책:** 각 Phase 완성 후 AI 에게 test_*.c 작성 위임 → 팀이 리뷰로 보강
- **근거:** 스프린트 1일이라는 제약. 구현 + 테스트를 사람이 다 쓰면 1.2배 시간 소요, AI 생성 + 리뷰는 1.0배 + α. 그 α 가 승패를 갈랐다.
- **한계:** AI 테스트는 정상/경계/에러 정도로 커버리지가 충분치 않았다. 특히 **concurrent / race 시나리오**는 사람이 직접 설계 — AI 로 다 해결된다고 착각하면 안 된다.

### 3.3 인터페이스 파일은 PM 단독 PR
- `include/bptree.h` 와 `include/types.h` 는 팀원 누구도 건드리지 않고 PM 단독 PR 로만 수정
- 근거: 인터페이스가 흔들리면 팀 전체가 blocked. **blast radius 가 큰 파일일수록 수정 권한을 좁혀야** 한다.

### 배운 것
협업 프로세스는 **팀 규모에 맞춰 단순해져야 한다.** CI/CD, automated merge queue 같은 건 필요 없었고, "PR 본문에 건드린 영역 명시 + PM mix merge + AI 테스트 위임" 만으로 충분했다.

## 4. 품질 — 타협하지 않은 3가지 게이트

| 게이트 | 근거 |
|---|---|
| `-Wall -Wextra -Wpedantic` 경고 0 | C 는 조용히 무너진다. 미사용 변수 · 암묵적 형변환이 1M 스케일에서 터진 뒤에는 추적 불가. |
| `valgrind --leak-check=full` 누수 0 | 1M 건 테스트라 누수 1바이트도 메가바이트로 불어남. |
| Week 6 기존 테스트 227개 회귀 0 | 인덱스 도입이 기존 기능을 깨면 안 됨 — "새 기능은 구 기능의 슈퍼셋" 원칙. |

### 배운 것
게이트를 "하면 좋은 것"이 아니라 **"PR 머지 조건"**으로 올리면 팀 전체 품질 하한선이 정해진다. 특히 valgrind 는 1일 스프린트에서 생략하고 싶은 유혹이 컸지만, 끝까지 지킨 덕에 발표 중 크래시 0.

## 5. 시연 — 의존성 0 웹 데모

### 시도
- Frontend: 정적 HTML + vanilla JS + **Chart.js CDN 한 줄**
- Backend: `python3 -m http.server` 계열 — **stdlib 만** (Flask/FastAPI X, npm X)
- 동작: `server.py` 가 `sqlparser --json` · `benchmark` 을 **subprocess** 로 호출 → stdout 파싱 → JSON 응답

### 근거
- **재현성:** 제3자가 `python3 web/server.py` 한 줄로 실행. npm install 대기 없음.
- **발표 안정성:** Docker 컨테이너 내에서도 Python 은 기본. pip 의존성 관리의 복잡도 = 발표 전 크래시 리스크.
- **C / Python 경계 명확:** sqlparser 는 C 바이너리, 서버는 프로세스 생성만. 언어 간 책임이 깔끔.
- **subprocess 모델과 lazy rebuild 의 궁합:** 각 요청마다 프로세스 새로 뜸 → 전역 상태 오염 없음 → `storage_ensure_index` 의 "첫 SELECT 에 재구성" 설계와 자연스럽게 맞음.

### 발표 시나리오: "새벽 3시 결제 장애"
- id 가 시간순 auto-increment → **id 범위 = 시간 구간 proxy**
- `WHERE id BETWEEN 30000 AND 31500` 을 1ms 내 반환 → 막대 차트로 "선형 대비 400배" 시각화
- 근거: 수치(400배)보다 **실제 도메인 상황극**이 사람 기억에 남는다.

### 배운 것
도구 선택은 **발표 품질을 결정한다.** 가장 화려한 프레임워크가 아니라 **"5분 안에 처음 보는 사람도 실행 가능한가"** 를 기준으로 골랐다. 의존성 0 원칙은 제약처럼 보이지만 사실 **자유**였다.

## 6. 되돌아보면 — 아쉬운 것과 다음에 해볼 것

### 아쉬운 것
- **`bptree_delete` 를 끝내 구현하지 않고 rebuild 방식으로 우회** — 안전했지만, DELETE 이후 인덱스 재구성 비용이 데이터 규모에 비례. 트리에서 직접 삭제하는 방식이 장기적으로 옳았다.
- **동시성 테스트 부재** — subprocess 모델 덕에 문제는 없었지만, 멀티스레드 환경이었다면 index_registry 의 선형 배열이 경합점이 됐을 것.
- **스키마 변경 시나리오 검증 없음** — 바이너리 fast path 는 고정폭이라 컬럼 추가 시 전체 재작성 필요. ALTER TABLE 대응은 미정.

### 다음에 해볼 것
- **Write-Ahead Log (WAL)** — BULK_INSERT_MODE 의 일반화. 지연 flush 를 크래시 안전하게.
- **mmap 기반 storage** — 파일 포인터 재사용의 극한. 커널 page cache 를 직접 공유.
- **io_uring 벤치** — syscall batching 을 OS 레이어에서 풀면 얼마나 더 빨라질지.
- **B+ Tree 에서 직접 삭제** — tombstone 방식 vs rebalance 방식 비교.

## 7. 총평 — 3줄 요약

1. **계약 고정 → 병렬 개발 → Mix merge** 라는 구조가 1일 스프린트의 엔진이었다.
2. **병목은 알고리즘이 아니라 I/O 경계**였고, 이 경계를 넘는 패턴은 DB · 웹 서버 · OS 전반에 걸쳐 동형이었다.
3. 제약(의존성 0, 1일, C 언어)은 선택지를 **좁혀준 덕에** 집중할 수 있었다. 제약이 없었다면 오히려 완성하지 못했을 것.
