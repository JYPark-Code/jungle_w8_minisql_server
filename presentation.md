# MiniSQL — Week 7: B+ Tree Index 구현


## 1. 프로젝트 목표

기존 시스템은 조건에 맞는 row를 찾을 때 **파일을 처음부터 끝까지 읽는 선형 탐색**을 사용했습니다.

데이터가 적을 때는 괜찮지만, 데이터가 많아질수록 비효율.

예)
```sql
SELECT * FROM users WHERE id = 999999;    -- 선형 탐색: 처음부터 다 읽음
SELECT * FROM payments WHERE id BETWEEN 30000 AND 31500;  -- 마찬가지
```

이번 프로젝트에서는 **id를 기준으로 B+ Tree 인덱스**를 도입해서 이 문제를 해결했습니다.

| 질의 | 기존 | 개선 후 |
|---|---|---|
| `WHERE id = ?` | O(n) 선형 탐색 | O(log n) 인덱스 탐색 |
| `WHERE id BETWEEN A AND B` | O(n) 선형 탐색 | O(log n + k) 범위 탐색 |

**결과: 순수 트리 탐색 기준 선형 대비 약 1,842배 빠름**

## 2. 아키텍처 및 시스템 흐름

```
사용자 SQL 입력
      │
      ▼
  parser.c          SQL 문장을 구조체(ParsedSQL)로 해석
      │
      ▼
  executor.c        어떤 경로로 실행할지 결정
      │
      ├── WHERE id = ? 또는 BETWEEN  →  B+ Tree 인덱스 조회 (빠름)
      │                                       │
      │                               id → row_index 위치 파악
      │                                       │
      └── 그 외 조건                 →  storage.c에서 실제 데이터 읽기
                                      (B+ Tree 경로도 최종적으로 여기서 읽음)
```

**핵심 설계 포인트**

- B+ Tree는 실제 row 데이터를 저장하지 않고, **`id → row_index` 매핑 정보만** 저장합니다.
- `index_registry`라는 전역 관리자가 모든 모듈이 **같은 트리 인스턴스를 공유**하도록 연결해줍니다.
- INSERT할 때 등록한 인덱스를 SELECT에서 그대로 재사용합니다.

## 3. 시연: 결제 로그 웹 데모

**시나리오: 새벽 3시 결제 장애 대응**

> id가 시간순으로 자동 증가하므로, id 범위 = 시간 구간 proxy

```sql
-- 장애 발생 구간의 결제 로그만 빠르게 조회
SELECT * FROM payments WHERE id BETWEEN 30000 AND 31500;
```

**데이터 모델**

```sql
CREATE TABLE payments (
    id         INT,     -- auto-increment (시간순)
    user_id    INT,
    amount     INT,
    status     TEXT,    -- 'SUCCESS' | 'FAIL' | 'TIMEOUT'
    created_at INT      -- Unix timestamp
);
```

**웹 UI 3가지 시나리오**

| 버튼 | 동작 |
|---|---|
| 더미 주입 | 10만~100만 건 결제 로그 생성 (실패율 5%, 타임아웃 2% 포함) |
| 장애 구간 조회 | `WHERE id BETWEEN` 으로 특정 구간 조회 |
| 선형 vs 인덱스 비교 | 같은 데이터를 두 방식으로 조회해 속도 비교 차트 출력 |

**성능 수치 (1M 행, `WHERE id BETWEEN 500000 AND 501500`)**

| 경로 | 소요 시간 |
|---|---|
| CSV 기존 방식 | 5.77 s |
| CSV + 고정폭 바이너리 fast path | **2.22 s** |

> 바이너리 fast path: N번째 행 접근 시 파일 전체를 읽지 않고 정확한 위치로 바로 이동(fseek)

## 4. 트러블슈팅 1 — INSERT 성능 최적화 (32 ms → 3 µs)

### 문제

초기 테스트에서 100만 건 삽입이 사실상 불가능한 수준이었습니다.

- 60초 타임아웃 안에 **2,716건**만 삽입 완료 (건당 32 ms)
- 추정 완료 시간: **약 8시간**

### 원인

알고리즘 문제가 아니었습니다. **개발 환경의 파일 시스템 레이어**가 원인이었습니다.

```
Windows Host
└── WSL2
    └── Docker Desktop
        └── devcontainer (우리 환경)
            └── 9p 프로토콜 + dirsync 마운트
```

이 구조에서는 `fopen`, `fclose`, `stat` 같은 파일 시스템 접근 하나하나가
**Windows 드라이브까지 왕복**해야 해서 1회당 약 0.3 ms가 소요됩니다.
row 하나를 INSERT할 때 이런 접근이 약 6번 → 건당 약 1.8 ms가 I/O에 소모됩니다.



### 해결: 5단계 최적화

| 단계 | 조치 | 효과 |
|---|---|---|
| ① 파일 포인터 재사용 | 매 row마다 열고 닫던 파일을 프로세스 동안 유지 + 64KB 버퍼 | 14.8 s → 13.5 s |
| ② 스키마 캐싱 | 매 INSERT마다 읽던 스키마를 처음 1회만 로드 후 재사용 | 13.5 s → 4.8 s |
| ③ 경로 캐싱 | 매번 탐색하던 파일 경로를 첫 호출 때 확정 후 재사용 | 4.8 s → 1.4 s |
| ④ meta 캐싱 | O(N²) 재스캔 제거 | O(N²) 제거 |
| ⑤ BULK_INSERT_MODE | 대량 삽입 시 fflush를 버퍼가 찰 때만 실행 | 1.4 s → **0.058 s** |

### 결과

| 모드 | 1M INSERT |
|---|---|
| 수정 전 | ~8시간 (추정) |
| 수정 후 (기본) | **138 s** |
| 수정 후 (BULK_INSERT_MODE) | **2.8 s** |

→ 건당 **32 ms → 3 µs**, 약 **10,000배** 개선

## 7. 정리

이번 프로젝트는 B+ Tree를 실제 SQL 엔진에 연결하면서 세 가지를 확인했습니다.

**① 인덱스가 왜 필요한가**
— 선형 탐색으로는 데이터가 늘어날수록 한계가 분명합니다. B+ Tree는 O(log n)으로 이를 해결합니다.

**② 실제 시스템에 어떻게 붙이는가**
— 자료구조를 구현하는 것과, 기존 SQL 실행기·저장소와 연결하는 것은 다른 문제입니다.
인터페이스 설계, index_registry, 정합성 유지 등 통합 과정의 문제를 풀었습니다.

**③ 성능은 알고리즘만의 문제가 아니다**
— 같은 코드라도 파일 시스템 레이어에 따라 성능이 100배 이상 차이납니다.
시스템 콜 횟수와 I/O 경로까지 함께 봐야 실제 성능 개선이 됩니다.
